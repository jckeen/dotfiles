### [HIGH-1] Unverified `curl | bash` Supply-Chain Escape Hatch in Setup Script
**File:** `setup.sh:413,418`
**CWE:** CWE-494 (Download of Code Without Integrity Check), CWE-829 (Inclusion of Functionality from Untrusted Control Sphere)

`setup.sh` installs bun using a SHA-256 pinned installer by default (positive), but provides an explicit unverified escape hatch via `BUN_UNPINNED=1`:

```bash
if [ "${BUN_UNPINNED:-0}" = "1" ]; then
  echo "  -> WARNING: BUN_UNPINNED=1 set; running unverified bun installer"
  run curl -fsSL https://bun.sh/install | bash
```

Any environment where `BUN_UNPINNED=1` is set (CI system, compromised `.bashrc`, inherited environment) executes an unverified script from the internet without confirmation. Attackers who can inject environment variables can silently disable integrity verification.

**Remediation:** Remove the `BUN_UNPINNED=1` bypass or make it require a positive interactive flag (`--force-unpinned`) rather than an environment variable. If the escape hatch must exist, require both the env var AND an explicit `--force-unpinned` CLI flag to activate.

### [MEDIUM-1] Hardcoded `localhost:31337` Unauthenticated Endpoint in PromptProcessing Hook
**File:** `claude/hooks/PromptProcessing.hook.ts:1010`
**CWE:** CWE-441 (Unintended Proxy/Intermediary), CWE-346 (Origin Validation Error)

The hook calls `fetch('http://localhost:31337/notify', ...)` with session names derived from prompt text. This is a plaintext HTTP POST to an unauthenticated local endpoint. On shared or multi-tenant hosts (including WSL2 with port forwarding), this endpoint is reachable from other processes. Any process binding port 31337 receives session name and voice configuration data from Claude sessions.

**Remediation:** Add a bearer token in the `Authorization` header. Consider using a Unix domain socket instead of a TCP port for inter-process communication. Document that the PAI voice server must be the exclusive listener.

### [MEDIUM-2] `HygieneStatus.hook.sh` Executes Hardcoded Absolute Path Without Validation
**File:** `claude/hooks/HygieneStatus.hook.sh`
**CWE:** CWE-426 (Untrusted Search Path), CWE-78 (OS Command Injection)

The hook executes `exec "$HOME/dev/dotfiles/hygiene-status.sh" --reminder` — hardcoding the dotfiles repo location. If `$HOME/dev/dotfiles` is a symlink pointing outside the expected location, the hook executes an arbitrary script.

**Remediation:**
```bash
TARGET="$(realpath "$HOME/dev/dotfiles/hygiene-status.sh" 2>/dev/null)"
[[ "$TARGET" == "$HOME/dev/dotfiles/"* ]] || exit 1
exec "$TARGET" --reminder
```

### [LOW-1] SHA-256 Pin in `setup.sh` Has No Automated Staleness Detection
**File:** `setup.sh:391–392`
**CWE:** CWE-494 (Download of Code Without Integrity Check)

The bun installer SHA-256 pin is correct practice, but pin rotation is a manual, undocumented process with no tooling to detect when the upstream installer has changed. A developer encountering a stale pin may reach for `BUN_UNPINNED=1` (see HIGH-1) to unblock themselves.

**Remediation:** Add a `check-bun-pin.sh` script that fetches the current installer and compares its digest against the pinned value. Integrate into CI so pin staleness is detected automatically.

### [LOW-2] ntfy Hook Sends Session Summaries to Public Server by Default
**File:** `claude/hooks/ntfy-awaiting-input.sh`
**CWE:** CWE-200 (Exposure of Sensitive Information), CWE-319 (Cleartext Transmission of Sensitive Information)

When `$NTFY_TOPIC` is set, this hook sends question headers and question text from AskUserQuestion events to `${NTFY_SERVER}/${NTFY_TOPIC}` — by default `ntfy.sh`, a public server. ntfy.sh topics are essentially public-readable by anyone who knows the topic name. Session context (project names, decisions, partial secrets in questions) may be exposed.

**Remediation:** Document clearly that `NTFY_TOPIC` should be a high-entropy unguessable string (32+ random characters). Consider a self-hosted ntfy instance. Add minimum-length validation for `NTFY_TOPIC` (16+ chars).

### [LOW-3] `PRWatcherAutoLaunch.hook.ts` — Confirm `repoSlug` Allowlist Validation
**File:** `claude/hooks/PRWatcherAutoLaunch.hook.ts:170,246–255`
**CWE:** CWE-78 (OS Command Injection — prior fix confirmation required)

Comments in the file acknowledge a prior injection vulnerability that was fixed by using `spawn` with argument arrays (correct mitigation). However, it is worth confirming that `parsePRUrl`'s output (`repoSlug`) is validated against an allowlist regex before being passed to `spawn`, and that the first argument to `spawn` is never a shell.

**Remediation:** Confirm `repoSlug` is validated against `^[A-Za-z0-9._-]+/[A-Za-z0-9._-]+$` before use. Add a unit test verifying that shell-metacharacter-laden repo slugs do not cause unexpected behavior.

### [INFO-1] `cc-functions.ps1` Self-Update Copies from Unverified WSL Source
**File:** `windows/cc-functions.ps1`
**CWE:** CWE-494 (Download of Code Without Integrity Check)

The `ccupdate` function copies the script from `\\wsl.localhost\...` (UNC path) to the Windows local copy without hash verification. If the WSL filesystem is compromised, this propagates the compromise to Windows PowerShell profile functions.

**Remediation:** Document the trust assumption explicitly. Consider adding a SHA-256 digest check of the WSL source against a stored baseline before copying.
